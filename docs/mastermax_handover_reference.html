تصحيح للمستند (دقة المستودع): ملف firebase.json الفعلي في الجذر يعرّف مصفوفة hosting بعدة أهداف: admin → build/web-admin، realestate → build/web-realestate، car → build/web-car — المقتطف أعلاه يوضّح الحد الأدنى (Firestore/Storage) فقط؛ راجع الملف الكامل عند النشر.

Firestore — القواعد في المستودع (محدَّث)

تحديث مارس 2026: أُعيدت كتابة firestore.rules (~198 سطر): قواعد لـ users (مع تمييز أدمن عبر isAdmin في المستند)، cars / properties (ملكية)، support_tickets، subscriptions، favorites، bank_transfers، verification_requests، مجموعات أخرى صريحة — لا يوجد match /{document=**} الواسع السابق فقط.

متبقٍّ تشغيليًا: نشر القواعد والفهارس إلى Firebase: firebase deploy --only firestore:rules,firestore:indexes؛ مراجعة سلوك الأدمن/الاستعلامات بعد النشر.

Storage — مقارنة (أكثر تقييدًا من Firestore حاليًا)

مثال من storage.rules: مسار الفيديوهات يسمح بالكتابة فقط إذا request.auth.uid == userId:

  allow read: if request.auth != null;
  allow write: if request.auth != null && request.auth.uid == userId;
}

07 الأمان والتشفير — إثبات من الكود

صنف SecurityEncryption (نمط singleton) يهيئ مفتاح AES عبر FlutterSecureStorage؛ الملف يستورد أيضًا crypto لعمليات التجزئة.

import 'dart:convert';
import 'dart:math';
import 'dart:typed_data';
import 'package:crypto/crypto.dart';
import 'package:encrypt/encrypt.dart' as encrypt;
import 'package:flutter_secure_storage/flutter_secure_storage.dart';

/// نظام تشفير البيانات الحساسة
class SecurityEncryption {
  static final SecurityEncryption _instance = SecurityEncryption._internal();
  factory SecurityEncryption() => _instance;

  static const String _masterKeyId = 'master_encryption_key';
  late final encrypt.Key _key;
  late final encrypt.IV _iv;

  final _secureStorage = const FlutterSecureStorage();

  SecurityEncryption._internal();

  Future<void> initialize() async {
    String? storedKey = await _secureStorage.read(key: _masterKeyId);
    if (storedKey == null) {
      final key = _generateSecureKey();
      await _secureStorage.write(key: _masterKeyId, value: base64.encode(key));
      storedKey = base64.encode(key);
    }
    _key = encrypt.Key(base64.decode(storedKey));
    _iv = encrypt.IV.fromSecureRandom(16);
  }

12.3 مسار موازٍ على الموبايل (يجب توحيده مع الويب)

12.4 المتبقّي والثغرات (أدلة سلوكية من الكود)

  • إعدادات التطبيق: الشاشة تذكر صراحةً إمكانية إضافة app_config لاحقًا — لا يوجد ربط Firestore بعد (app_settings_screen.dart 38–40).
  • فيديوهات يتيمة (ويب): نص يوضح الاعتماد على الموبايل أو Cloud Function (orphaned_videos_screen.dart 29–31).
  • تسجيل دخول الأدمن: admin_login_screen.dart — في release لا تُعبَّأ الحقول؛ في debug فقط يمكن تمرير بريد عبر --dart-define=ADMIN_LOGIN_DEBUG_EMAIL=… (لا كلمة مرور في الكود).

تحذير: وجود كلمات مرور/بريد في سكربتات التهيئة أو أدوات الهجرة (مثل data_migrator.dart) يتطلب مراجعة يدوية وإزالة من المستودع العام — لا تُلصق أسرار في هذا الملف HTML.

15 تحويل الباكند: من Firebase إلى REST + Swagger (OpenAPI)

تصحيح مفاهيمي: Swagger / OpenAPI ليست قاعدة بيانات ولا بديل Firebase جاهز — هي وثيقة وواجهة توثيق لواجهة REST (مسارات HTTP + نماذج JSON + تفويض). التحويل الحقيقي هو: Firebase (Auth + Firestore + Storage + …) ← خادم تملكه + قاعدة بيانات + تخزين ملفات، ويُعرَّف العقد عبر OpenAPI ويُختبر من Swagger UI.

وثيقة Swagger التي تعرضها (مثلاً «My Order API» لخدمات طعام/تموين) مجال منتج مختلف عن MasterMax (عقارات، سيارات، Spotlight). أي خادم جديد يحتاج عقد API خاصًا بكيانات المشروع (cars، properties، users، …) وليس نسخ مسارات مشروع آخر.

15.3 ماذا هو الأنسب لهذا المشروع؟ (رأي تقني مُبرَّر)

15.6 خطوات العمل بالتفصيل (مع ربط بالمستودع)

  1. جرد API ضمني: استخراج أسماء مجموعات Firestore والعمليات من الخدمات — مجلدات مثل lib/src/features/*/services/ وlib/admin_web/services/.
  2. نمذجة OpenAPI: تعريف مسارات REST (مثل GET/POST /api/cars، /api/properties، /api/auth/login) متوافقة مع نماذج Flutter الحالية (car_model.dart، property_model.dart، …).
  3. خادم وقاعدة بيانات: اختيار إطار (Node/Nest، .NET، Go، …) + PostgreSQL أو ما يناسب؛ تنفيذ Auth (Access + Refresh) يعادل سلوك auth_service.dart.
  4. التخزين: استبدال روابط firebase_storage بتوقيعات URL من الخادم أو تخزين كائنات (S3-compatible) — مراجعة car_image_service.dart، property_image_service.dart، video_service.dart.
  5. اللحظية: استبدال snapshots() بـ WebSocket/SSE أو اختيار إبقاء Firestore مؤقتًا لقنوات معيّنة فقط (هجرة هجينة).
  6. طبقة عميل Flutter: إدخال ApiClient (مثل dio الموجود في المشروع) واستبدال استدعاءات FirebaseFirestore.instance تدريجيًا داخل كل Service؛ الحفاظ على واجهات Provider قدر الإمكان لتقليل تغيير الشاشات.
  7. الأدمن والويب: تعديل admin_web لنفس الـ API؛ إزالة افتراض أن كل شيء في Firestore مباشرة من المتصفح إن انتقل التفويض للخادم.
  8. الإشعارات: استبدال تكامل FCM بخادم يرسل عبر FCM HTTP v1 أو مزود آخر — مراجعة firebase_messaging في pubspec.yaml.
  9. الهجرة: سكربتات تصدير من Firestore + استيراد مع تحقق؛ تشغيل مزدوج لفترة؛ إيقاف الكتابة على Firebase بعد التحقق.
  10. Swagger UI: نشر وثيقة OpenAPI على بيئة داخلية؛ ربطها ببيئات dev/staging — للاختبار اليدوي وليس كبديل تنفيذ.

15.8 إطلاق خلال شهر — لا تُحوّل الباكند الآن

الوضع ليس «صعبًا بالضرورة» للإطلاق: الأشهر الطويلة في القسم 15 كانت لتقدير تحويل كامل من Firebase إلى خادم REST. هذا غير مطلوب لإصدار أول خلال هذا الشهر.

ما الذي تفعلونه هذا الشهر (واقعي):

  • ابقوا على Firebase — التطبيق مبني عليه؛ الإطلاق السريع يعني تثبيت الجودة والأمن على ما هو موجود.
  • الحد الأدنى من الأمن قبل المستخدمين: تشديد firestore.rules قدر الإمكان (حتى لو على أهم المجموعات أولًا)؛ إزالة الأسرار من المستودع؛ إصلاح payment_processor.dart (معرّف المستخدم)؛ إزالة تعبئة دخول الأدمن التجريبية في الإنتاج — انظر القسم 14 (بنود P0/P1).
  • فهارس Firestore عند ظهور أخطاء failed-precondition في الأدمن أو الويب — إنشاء الفهرس من رابط الكونسول (دقائق إلى ساعات، لا أشهر).
  • اختبار مسارات حرجة: تسجيل، إضافة إعلان، دفع/اشتراك إن كان مفعّلًا، ويب السيارات/العقارات حسب النطاق.
  • تأجيل «التحويل إلى Swagger/REST» إلى بعد الإطلاق — كمرحلة منتج منفصلة عندما يكون لديكم وقت وفريق خلفية.

الخلاصة: الإطلاق هذا الشهر ممكن مع Firebase؛ الخطر كان في الخلط بين «إطلاق المنتج» و«إعادة بناء الباكند بالكامل». الثانية تُؤجَّل؛ الأولى تُنجَز بقائمة تحقق قصيرة من القسم 14.
